## ðŸŽ“ Key Concepts

### The Two-Role Architecture

Every agent built with this protocol has **exactly 2 roles**:

#### **Role 1: Agent Builder (System Role)** - MANDATORY

**Always included**, standardized across all projects:

- **Purpose**: System maintenance, KG evolution, self-improvement
- **Always active**: Yes (runs in background)
- **Skills**: KG maintenance (monitor, update, heal), validation
- **Templates**: 100% reusable (copy from protocol)
- **Customization**: None needed

#### **Role 2: [Your Domain Role] (Custom Role)** - CUSTOMIZED

**Tailored to your project**:

- **Purpose**: Domain-specific tasks (your use case)
- **Activated**: On user request
- **Skills**: Your domain capabilities
- **Templates**: Used as guides, adapted to domain
- **Customization**: Extensive (based on Phase 1-2 design)

**Why this pattern**:
- âœ… **Reliability**: Agent Builder is tested and standardized
- âœ… **Flexibility**: Domain role adapts to any use case
- âœ… **Separation**: System vs domain concerns
- âœ… **Self-evolution**: Every agent can maintain itself

### Fully-Connected Knowledge Graphs

**The Core Innovation**:

Every Knowledge Graph node (except root) has a `parent` field:

```json
{
  "id": "sys:docs:builder:arch",
  "type": "folder",
  "title": "Architecture Documentation",
  "parent": "sys:docs:builder",  â† Required!
  "path": "docs/builder/architecture/"
}
```

**Plus cross-graph links**:

```json
{
  "id": "behavior:agent-builder:kg-consistency",
  "links": {
    "enforces": ["skill-maintain-kg"],  â† Cross-graph relationship
    "governs": ["sys:agents"]
  }
}
```

**Result**: Agent can navigate from ANY node to ANY related node (true "Mind GPS")

### Self-Evolution Capability

**Built-in KG maintenance**:

1. **Monitor** (rule-based): Detect file changes
2. **Update** (hybrid): Interpret changes, update KG
3. **Heal** (rule-based): Fix broken links

**When user says**: "Update the knowledge graph"

**Agent does**:
1. Scans file system for changes
2. Generates metadata for new files (LLM)
3. Updates JSON graphs (rules)
4. Validates connectivity (rules)
5. Confirms success

**No manual KG editing needed!**
